Подготовить и обосновать тестовые сценарии для:

- позитивного тестирования;
- негативного тестирования;

## 2. Объект тестирования

Приложение `GymManagementClient` представляет собой WinForms-клиент на `.NET Framework 4.8` с подключением к PostgreSQL.

Ключевые участки кода, использованные как тестовая база:

- запуск приложения: `Program.cs:13`;
- аутентификация и валидация логина: `Forms/LoginForm.cs:33`;
- обработка типовых UI-ошибок: `UI/LiquidGlassForm.cs:27`;
- открытие рабочих форм из главного окна: `Forms/MainForm.cs:103`;
- регистрация клиента на занятие: `Forms/RegisterSessionForm.cs:34`;
- статистика тренеров и переход к деталям: `Forms/TrainerStatsForm.cs:37`;
- проверка наличия строки подключения: `Data/DatabaseClient.cs:11`.

## 3. Среда и методика

Дата выполнения: `23.04.2026`.

Среда:

- ОС Windows;
- локальная сборка `bin\\Debug\\GymManagementClient.exe`;
- PostgreSQL на `localhost:5432` доступен по сети;
- сборка подтверждена командой `dotnet msbuild GymManagementClient.csproj /t:Build /p:Configuration=Debug`.

Методика:

- статический анализ кода для выбора зон риска;
- запуск приложения как отдельного процесса;
- автоматизированные проверки через PowerShell и reflection для воспроизведения UI-веток без ручного ввода;
- изолированные проверки на копиях каталога приложения во временных папках.

Ограничение стенда:

- в репозитории отсутствуют подтвержденные валидные учетные данные БД, поэтому полный позитивный сценарий "успешный вход + работа с реальными данными БД" не подтверждался end-to-end;
- часть негативных сценариев по БД выполнялась через автоматизированный вызов обработчиков формы, чтобы проверить реакцию приложения на сбой, а не бизнес-корректность авторизации.

## 4. Обоснование набора сценариев

### 4.1. Позитивные сценарии

Позитивные проверки нужны, чтобы подтвердить базовую работоспособность приложения и корректность стартового пользовательского пути.

- `FT-01`: запуск приложения из штатного каталога сборки.
  Обоснование: это минимальный базовый сценарий эксплуатации.
- `FT-02`: генерация капчи при открытии формы входа.
  Обоснование: капча обязательна для дальнейшего входа, поэтому стартовая логика формы должна отрабатывать стабильно.

### 4.2. Негативные сценарии

Негативные проверки нужны, чтобы убедиться, что ошибочный ввод не приводит к сбою и сопровождается понятным сообщением.

- `FT-03`: попытка входа с пустыми полями.
  Обоснование: базовая проверка обязательных полей.
- `FT-05`: попытка перехода в ветку подключения к БД при ошибочном входе.
  Обоснование: необходимо убедиться, что сбой при подключении не валит форму входа мгновенно.

### 4.3. Проверки несоответствия типов данных

В коде явно присутствуют преобразования `int.TryParse` и `Convert.ToInt32`, поэтому ошибки типов являются реальным риском.

- `FT-04`: ввод буквенного значения в поле капчи.
  Обоснование: поле ожидает целое число.
- `FT-06`: подмена `SelectedValue` в форме записи на занятие строковыми значениями вместо `Int32`.
  Обоснование: в `Forms/RegisterSessionForm.cs:47` выполняется `Convert.ToInt32`.
- `FT-11`: передача строкового `trainer_id` в форме статистики тренеров.
  Обоснование: в `Forms/TrainerStatsForm.cs:58` выполняется прямое преобразование без локального `try/catch`.

### 4.4. Проверки перемещения файлов и папки приложения

Для настольного клиента критично проверить, сохраняет ли он работоспособность после копирования/переноса и что происходит при нарушении состава файлов.

- `FT-07`: перенос всей папки приложения.
  Обоснование: типовой сценарий развертывания или переноса на другой каталог.
- `FT-08`: перенос приложения без файла `GymManagementClient.exe.config`.
  Обоснование: проверка зависимости от файла конфигурации.
- `FT-09`: перенос приложения без `Npgsql.dll`.
  Обоснование: проверка реакции на отсутствие критической библиотеки.
- `FT-12`: обращение к логике входа после удаления `Npgsql.dll`.
  Обоснование: важно проверить не только запуск окна, но и момент первого обращения к БД.

### 4.5. Проверки исключений и аварийного завершения

В `Program.cs:13` нет глобального перехвата `Application.ThreadException` и `AppDomain.CurrentDomain.UnhandledException`, поэтому нужно проверить реальные crash-сценарии.

- `FT-10`: открытие формы клиентов без строки подключения.
  Обоснование: `Data/DatabaseClient.cs:11` бросает `InvalidOperationException`, а `Forms/MainForm.cs:103` открывает форму без защитного `try/catch`.
- `FT-11`: двойной щелчок по статистике тренеров при неверном типе `trainer_id`.
  Обоснование: прямое преобразование в `Int32` без локального перехвата может аварийно завершить приложение.

## 7. Выявленные проблемы

### Дефект 1. Отсутствует глобальный перехват необработанных исключений

Подтверждение:

- в `Program.cs:13` приложение запускается напрямую через `Application.Run(new LoginForm());`;
- подписки на `Application.ThreadException` и `AppDomain.CurrentDomain.UnhandledException` отсутствуют.

Риск:

- любое исключение вне `TryRunUiAction(...)` приведет к аварийному завершению приложения.

### Дефект 2. Открытие DB-зависимых форм из `MainForm` может аварийно завершить приложение

Подтверждение:

- `Forms/MainForm.cs:103` создает `new ClientsForm()` без локального `try/catch`;
- `Data/DatabaseClient.cs:11` выбрасывает `InvalidOperationException`, если строка подключения пуста;
- сценарий `FT-10` это воспроизвел.

Риск:

- при потере сессии или неконсистентном `SessionContext` пользователь получает crash вместо управляемой ошибки.

### Дефект 3. В `TrainerStatsForm` неверный тип `trainer_id` вызывает необработанное исключение

Подтверждение:

- в `Forms/TrainerStatsForm.cs:58` используется `Convert.ToInt32(cellValue)` без локального перехвата;
- сценарий `FT-11` воспроизвел `System.FormatException`.

Риск:

- поврежденные данные таблицы или несоответствие схемы представления могут завершить приложение аварийно.

### Дефект 4. Отсутствие `Npgsql.dll` не выявляется при старте и приводит к отказу только при первом доступе к БД

Подтверждение:

- сценарий `FT-09`: приложение стартует даже после удаления `Npgsql.dll`;
- сценарий `FT-12`: при первом вызове логики входа возникает `FileNotFoundException`.

Риск:

- дефект поставки проявится не сразу, а уже в пользовательской работе, что ухудшает диагностируемость и повышает риск аварийного отказа.

## 8. Итоговый вывод

Базовая оболочка приложения работает: запуск, отображение формы входа, генерация капчи, первичная валидация обязательных полей и часть защит от неверных типов реализованы корректно.

При этом устойчивость приложения к исключительным ситуациям недостаточна:

- нет глобального перехвата необработанных исключений;
- часть DB-зависимых веток открывается без защитного слоя;
- неверный тип данных в статистике тренеров приводит к `FormatException`;
- нарушение состава файлов поставки (`Npgsql.dll`) не выявляется при запуске и проявляется только в момент обращения к БД.

## 9. Рекомендации

1. Добавить в `Program.Main` глобальные обработчики `Application.ThreadException` и `AppDomain.CurrentDomain.UnhandledException` с записью в лог и пользовательским сообщением.
2. Оборачивать создание DB-зависимых форм из `MainForm` в защищенный вызов с показом управляемой ошибки.
3. В `TrainerStatsForm` заменить прямой `Convert.ToInt32` на безопасное преобразование с сообщением пользователю.
4. Добавить preflight-проверку состава поставки на старте: наличие `Npgsql.dll`, конфигурации и обязательных зависимостей.
5. Для полноценного позитивного тестирования подготовить отдельный тестовый стенд с известными валидными учетными данными и тестовыми данными БД.
